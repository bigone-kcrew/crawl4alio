# 파이프라인 안정화 노트 — 대용량 적재와 단일 파일 인덱스에서 배운 것들

2026-07-17 하루 동안 실제 운영(155,113문서 / 1,208,291청크 규모)에서 겪은 사고들과,
그 뒤에 코드·절차로 굳힌 원칙을 정리한다. 같은 규모의 데이터를 다루는 사람이
같은 함정을 다시 밟지 않도록 사고 경위까지 남겨둔다.

## 1. 대용량 JSONL 쓰기: 동기 루프 + WriteStream은 폭탄이다

**사고**: 파서가 수십만 문서를 도는 완전 동기 루프 안에서 `stream.write()`만 호출했다.
동기 루프 중에는 이벤트 루프가 한 번도 돌지 않으므로 flush가 일어나지 않고,
산출물 수 GB가 통째로 프로세스 메모리의 스트림 버퍼에 쌓였다가 루프 종료 후
한꺼번에 writev로 쏟아진다. 이 대형 flush가 실패하면서 프로세스가 죽었고,
파일은 **정확히 4GiB(2³²)에서 절단**된 채 60.7만 줄이 중복 기록돼 있었다.

**원칙**:
- `write()`가 false를 반환하면 반드시 `drain`을 기다린다 (`events.once(stream, 'drain')`).
- 스트림에 `error` 핸들러를 달고, 실패하면 **반드시 비정상 종료**한다.
  조용히 죽으면 후속 적재 단계가 불완전한 스테이징을 읽는다.
- `end()`의 완료 콜백을 기다린 뒤에야 요약 리포트를 쓴다. 리포트가 먼저 쓰이면
  "통계는 정상인데 파일은 깨진" 상태가 되어 탐지가 늦어진다.

적용: `rag/parse_disclosure.js` (v1.9.1).

## 2. PostgreSQL TEXT는 NUL(0x00) 한 글자에 전체 적재가 무너진다

**사고**: OCR/변환 산출물 md 한 건에 NUL 문자가 섞여 있었다. 121만 청크 적재가
40만 지점에서 `invalid byte sequence for encoding "UTF8": 0x00`으로 중단됐다.
오염원은 **전체 121만 청크 중 단 1건**이었다.

**원칙**:
- 파서가 쓰는 모든 텍스트 필드에서 NUL을 제거한다 (`.replace(/\u0000/g, '')`).
  PG의 TEXT가 거부하는 문자는 NUL뿐이므로 다른 제어문자까지 지울 필요는 없다
  (지우고 싶다면 검색 품질 관점에서 별도로 판단).
- 적재 전에 `rag/validate_staging.js <corpus>` 를 게이트로 돌린다. 라인 JSON 유효성,
  (doc_id, seq) 유일성, docs의 n_articles 합계 대조, NUL/제어문자 검사를 한 번에 한다.
  위 두 사고 모두 이 게이트가 있었으면 적재 전에 잡혔다.

## 3. `명령 | tee 로그` 뒤의 `$?`는 tee의 성공을 본다

**사고**: 파이프라인 스크립트의 실패 게이트가 전부 `node ... | tee -a $LOG; rc=$?`
꼴이었다. 파이프라인의 `$?`는 마지막 명령(tee)의 종료코드라 **게이트가 한 번도
작동한 적이 없었고**, 파싱이 죽어도 TRUNCATE 적재가 그대로 진행됐다.

**원칙**:
- 스크립트 머리에 `set -o pipefail` (bash). 이때부터 `$?`가 실패한 명령의 rc가 된다.
- 각 단계 뒤에 rc 게이트를 두고, 특히 **TRUNCATE·DELETE가 뒤따르는 단계 앞은 필수**.
- 이미 떠서 돌고 있는 셸 스크립트는 파일을 고쳐도 소용없다(시작 시점 버전으로 돈다).
  장시간 파이프라인을 고쳤다면 다음 실행부터 적용된다는 걸 기억할 것.

## 4. 단일 파일 HTML 인덱스: 데이터가 커지면 gzip 인라인 + async 해제

50MB에 육박하는 인라인 JSON을 품은 index.html은 열리긴 하지만 전송·파싱이 느리다.
데이터를 `zlib.gzipSync(...).toString('base64')`로 임베드하고 브라우저에서
`DecompressionStream('gzip')`으로 푸는 방식으로 46MB → 4.2MB가 됐다.
외부 의존이 없어 file:// 더블클릭 실행과 압축 배포가 그대로 유지된다.
(요구 브라우저: Chrome/Edge 2020+, Firefox 113+, Safari 16.4+ — 미지원 안내문 필수)

**함정 — async 로딩으로 바꾸는 순간 톱레벨 파생 계산이 전부 빈 데이터를 본다.**
`const MAP = {}; for (const x of DATA) ...` 같은 코드가 스크립트 파싱 시점에 실행되면
DATA는 아직 빈 배열이다. 페이지는 멀쩡히 뜨는데 특정 탭·목록만 비는, 원인을 눈치채기
어려운 증상이 된다. **파생 계산은 전부 함수로 감싸고 로딩 완료 후 부트에서 호출**한다.

**검증 — grep과 육안으로는 못 잡는다.** Node 18+는 atob/Blob/Response/DecompressionStream을
내장하므로, vm 샌드박스 + 최소 DOM 목으로 생성된 HTML을 실제로 부팅시켜
컨테이너에 콘텐츠가 렌더되는지 확인할 수 있다. 이때 innerHTML 길이만 보면
appendChild로 그리는 렌더러를 놓치므로 **자식 트리까지 가중치로 세야** 한다.

## 5. 코드(코드번호)의 의미를 추측으로 사전화하지 말 것

공시 항목 코드에 사람이 라벨을 달아둔 하드코딩 사전이 있었는데, 8개가 실제와
달랐다(예: "신규 채용 정보"라고 적힌 코드의 실제 내용은 징계제도 운영현황).
보름 넘게 아무도 몰랐고, 사용자가 "카드를 눌렀는데 다른 내용이 나온다"고
보고하고서야 발각됐다.

**원칙**: 라벨·분류는 공식 카탈로그(`data/disclosure_items.json`)와
**실제 파일명 표본**을 대조해서 데이터 주도로 생성한다. 사람이 손으로 쓴 사전은
초기 부트스트랩일 뿐, 검증 없이 신뢰하지 않는다. 대조는 코드별로
(큐레이션 라벨, 카탈로그 item_name, 폴더명, 파일명 상위 빈도) 네 값을 나란히 놓으면
불일치가 한눈에 드러난다.

## 6. 자잘하지만 아픈 것들

- **`pkill -f 패턴`은 자기 자신을 죽인다** — 호출한 셸의 커맨드라인에도 패턴이 있기
  때문. `pkill -f "이름[a-z]"` 처럼 문자클래스로 자기매칭을 깨거나 pgrep으로 PID를
  추려서 죽인다.
- 파일명에서 날짜를 못 뽑았다고 항목을 버리지 말 것 — 연도 폴더, 메타데이터
  (`disclosure_no` 앞 8자리 = 공시 등록일) 같은 폴백이 대개 존재한다.
- `grep`으로 이스케이프 시퀀스(`\u0000`)를 찾는 건 셸 인용 때문에 오탐·미탐이 잦다.
  JSON은 파싱해서 실제 문자열을 검사하는 쪽이 확실하다 (`validate_staging.js`가 그 방식).
